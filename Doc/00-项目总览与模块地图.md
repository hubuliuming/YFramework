# YFramework 项目总览与模块地图

## 目的

这组文档不是面向最终用户的教程，而是面向 AI 与维护者的“代码地图”。
目标是帮助阅读者在进入项目时，快速判断：

- 哪些目录是主干代码。
- 哪些能力是可复用基础设施。
- 哪些脚本只是示例、遗留方案或占位文件。
- 当需要改某个功能时，应该先看哪一块。

## 本次扫描范围

本次梳理主要覆盖以下脚本区域：

- `Assets/YFramework`：主干框架代码，约 68 个脚本。
- `Assets/Old`：遗留模块，约 6 个脚本。
- `Assets/Tests`：示例与验证脚本，约 8 个脚本。

默认不作为主干文档对象的目录：

- `Assets/Plugins`
- `Assets/TextMesh Pro`
- `Assets/.qframework`
- `Library`
- `Temp`
- `obj`

## 一句话结论

`YFramework` 当前更像一套 Unity 工具型基础库，而不是完整的业务框架。
它的核心特征是：

- 用 `YMonoBehaviour` 统一一部分组件约定。
- 用 `AutoBind` 自动生成 Designer 字段绑定。
- 提供一批常用的 `Kit`、`Extension`、`UI`、`Editor` 工具。
- 遗留资源加载/Reader 方案单独放在 `Assets/Old`。

## 模块地图

| 区域 | 目录 | 角色 | 重点关注 |
| --- | --- | --- | --- |
| 框架基座 | `Assets/YFramework/Framework` | 统一运行时基类与 AutoBind 规则定义 | `YMonoBehaviour`、`MonoGlobal`、`AutoBindRules` |
| AutoBind 生成链 | `Assets/YFramework/Framework/AutoBindE` + `Assets/YFramework/Editor/AutoBindE` | 自动生成 `*.Designer.cs` 并回填引用 | `AutoBindEditor`、`AutoBindFieldAttribute` |
| 通用能力库 | `Assets/YFramework/Kit` | 协程延迟、计时器、网络、单例、点击事件、文件与工具方法 | `ActionKit`、`TimerManager`、`TcpClient`、`MonoSingleton` |
| 扩展方法库 | `Assets/YFramework/Extension` | 对 `Transform`、`GameObject`、字符串、集合、动画等做扩展 | `TransformExtension`、`GameObjectExtension` |
| 数据结构与数学 | `Assets/YFramework/Collections` + `Assets/YFramework/Math` | Unity 可序列化容器与定点数实现 | `SerializableDictionary`、`Fixed64` |
| UI 系统 | `Assets/YFramework/UI` | UI 基类、显示栈、自定义 Image、布局与事件辅助 | `UIBase`、`UIManager`、`CircleImage` |
| 编辑器工具 | `Assets/YFramework/Editor` + `Assets/YFramework/UI/Editor` | 自动保存、导入规则、资源命名检查、UI 菜单项与自定义 Inspector | `AutoSaveWindow`、`InputResourcesSetting` |
| 遗留模块 | `Assets/Old` | 旧 Reader/资源加载方案 | `ReaderManager`、`ResLoader` |
| 测试/样例 | `Assets/Tests` | AutoBind 生成示例、输入测试、临时验证脚本 | `Test1`、`Test2`、`Test3` |

## AI 阅读时的全局约定

### 1. 默认主干只有 `Assets/YFramework`

做新功能、修 bug、补文档时，默认先从 `Assets/YFramework` 找答案。
`Assets/Old` 只作为历史实现参考，不应默认继续扩展。

### 2. 源码头注释有乱码时，以类型名和实际代码为准

项目中不少文件头部中文注释存在编码问题。
因此阅读优先级应该是：

1. 目录位置
2. 类型名 / 方法名 / 字段名
3. 实际调用关系
4. 最后才参考注释文本

### 3. 不要把占位类当成已完成模块

以下文件当前更像占位或未完成入口：

- `Assets/YFramework/Msg/Msg.cs`
- `Assets/YFramework/Kit/Time/TimerKit.cs`
- `Assets/YFramework/Kit/Action/ActionSpan.cs`

这些文件可以在文档和设计层被提到，但不要假设它们已经承担完整职责。

### 4. AutoBind 是项目最有辨识度的一条主线

如果问题涉及：

- 场景对象字段自动绑定
- `partial class`
- `*.Designer.cs`
- 子节点命名规则
- `ParentMono`

那么应优先阅读 `02-AutoBind机制.md`。

### 5. UI 不是完整窗口框架，更像轻量显示栈 + 工具集

`UIBase` / `UIManager` 提供的是简单显示与回退栈，不是大型页面系统。
如果业务要加复杂面板流转，通常需要在现有基础上继续封装。

## 本次文档拆分理由

没有按文件夹一比一地机械拆文档，而是按“AI 真正会问的问题”组织：

- 基座怎么工作？
- AutoBind 到底怎么生成和绑定？
- 通用工具都在哪？
- UI 应该从哪几个类入手？
- 编辑器工具和运行时代码如何区分？
- 哪些代码不要当成主干？

这样的文档结构更适合后续做：

- AI 上下文喂给
- 新人入门
- 代码检索
- 模块边界判断
- 重构前认知对齐
