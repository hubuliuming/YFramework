# 核心运行时与框架基座

## 范围

本篇主要覆盖：

- `Assets/YFramework/Framework`
- 与运行时基座强相关的全局入口

重点文件：

- `Assets/YFramework/Framework/YMonoBehaviour.cs`
- `Assets/YFramework/Framework/MonoGlobal.cs`
- `Assets/YFramework/Framework/AutoBindE/AutoBindRules.cs`

## 整体定位

YFramework 的“框架基座”非常轻。
它没有完整的应用层、模块注册器或复杂生命周期编排，而是提供：

- 一个统一基类 `YMonoBehaviour`
- 一个全局 Mono 容器 `MonoGlobal`
- 一套 `AutoBind` 命名与过滤规则

换句话说，这里更像“项目约定的地基”，不是完整的业务框架骨架。

## `YMonoBehaviour`

### 角色

`YMonoBehaviour` 是项目里最核心的运行时基类。
它同时实现了 `IAutoBindMono`，因此很多脚本一旦继承它，就天然参与 `AutoBind` 体系。

### 主要成员

- `public virtual void OnAwake()`
- `public abstract void OnStart()`
- `public MonoBehaviour MonoSelf => this`
- `public virtual bool IgnoreSelf { get; } = false`

### 关键理解

`YMonoBehaviour` 不是 Unity 生命周期的完整封装。
它只是定义了 `OnAwake` / `OnStart` 这两个约定入口，但**没有在基类内部自动调用 Unity 的 `Awake` / `Start`**。

这意味着：

- 子类如果希望 `OnStart` 生效，需要自己在 Unity 的 `Start` 里调用它，或者自行重写生命周期。
- `OnAwake` 目前更像约定接口，而不是已经接入的统一回调。

在 `Assets/Tests/Scripts/Test3.cs` 中可以看到典型用法：显式在 `Start()` 中调用 `OnStart()`。

### 与 AutoBind 的关系

`AutoBindRules.FiltrationElementTypes` 默认包含 `YFramework.YMonoBehaviour`。
因此在 AutoBind 递归扫描子节点时，挂了 `YMonoBehaviour` 子类的对象会被继续视作可递归绑定节点。

## `IAutoBindMono`

### 角色

`IAutoBindMono` 是 AutoBind 能识别的运行时接口。
`YMonoBehaviour` 已经实现它，但普通 `MonoBehaviour` 也可以自己实现。

### 主要约定

- `MonoSelf`：返回当前实际 Mono 实例。
- `IgnoreSelf`：是否忽略该节点，避免被父级递归 AutoBind 继续处理。

### 使用意义

如果业务类不想继承 `YMonoBehaviour`，但又想接入 AutoBind，可以直接实现 `IAutoBindMono`。

## `MonoGlobal`

### 角色

`MonoGlobal` 是一个懒加载的、`DontDestroyOnLoad` 的全局 MonoBehaviour 容器。

### 主要行为

- 首次访问 `MonoGlobal.Instance` 时，如果不存在，就会创建一个名为 `MonoGlobal` 的 `GameObject`。
- 在 `Awake()` 中调用 `DontDestroyOnLoad(gameObject)`。

### 典型用途

它主要被当作“协程宿主”使用。
例如 `ActionKit.Delay()` 和 `ActionKit.DelayOneFrame()` 最终都是把协程挂到 `MonoGlobal.Instance` 上启动。

### 影响

这让很多静态工具方法可以在没有业务 Mono 引用时依旧启动协程。
代价是：

- 运行时会自动生成全局节点。
- 某些静态调用会隐式创建对象。

## 基座层的运行时风格

从当前实现看，项目更偏向下面这类风格：

- 轻基类
- 静态工具方法
- 需要时再生成管理器
- 用单例而不是显式注入

典型例子：

- `MonoGlobal.Instance`
- `TimerManager.Instance`
- `UIManager.Instance`

## 与 `Kit` 的边界

基座层负责定义最低层约定；
真正的工具实现大量放在 `Kit` 中。

例如：

- 单例实现在 `Assets/YFramework/Kit/Singleton`
- 定时器实现在 `Assets/YFramework/Kit/Time`
- 网络实现在 `Assets/YFramework/Kit/Net`

因此理解方式应该是：

- `Framework` 定义规则和基础入口。
- `Kit` 提供具体能力。

## 当前设计里的注意点

### 1. `OnStart` 不是自动执行

这是整个框架最容易误判的地方之一。
不要因为看到抽象方法就默认它会像 Unity `Start()` 一样被统一调度。

### 2. `OnAwake` 目前也只是约定

源码里没有统一桥接逻辑。
如果业务依赖它，必须在子类里自行接入。

### 3. 继承 `YMonoBehaviour` 等于默认接入 AutoBind 生态

这对 UI、组件树递归绑定很方便，但也意味着：

- 子节点是否被递归处理，要关注 `IgnoreSelf`
- 命名、生成 Designer、字段回填这些规则都会跟着生效

### 4. 当前不存在真正成型的消息总线

`Assets/YFramework/Msg/Msg.cs` 目前是空壳。
不要把它当成一个已经可用的事件系统或消息总线。

## AI 修改建议

如果你要改动运行时基座，优先先判断以下几点：

- 是不是只是想加一个通用工具？如果是，通常应该放在 `Kit` 或 `Extension`。
- 是否真的需要改 `YMonoBehaviour`？改这里会影响 AutoBind 递归过滤。
- 是否会引入新的隐式对象创建？如果会，要注意和 `MonoGlobal` / 单例风格保持一致。
- 是否会让 `Start` / `Awake` 语义更混乱？当前基座已经存在“约定方法不自动执行”的特点，修改时应特别谨慎。
