# 编辑器工具

## 范围

本篇覆盖：

- `Assets/YFramework/Editor`
- `Assets/YFramework/UI/Editor`

这些代码运行在 Unity Editor 中，不属于玩家运行时逻辑。

## 总体定位

YFramework 的编辑器层主要做三类事：

- 自动化生产或生成代码
- 资源导入与命名约束
- 提升 UI / 场景编辑效率

其中最核心的一类已经单独成篇：

- `02-AutoBind机制.md`

本篇更多关注 AutoBind 之外的编辑器功能。

## 1. 场景自动保存

文件：

- `Assets/YFramework/Editor/AutoSaveScene/AutoSaveWindow.cs`
- `Assets/YFramework/Editor/AutoSaveScene/XPAutoSave.cs`

### `AutoSaveWindow`

提供一个简单的 EditorWindow，用来配置：

- 是否自动保存
- 时间间隔
- 是否输出消息

入口菜单：

- `YFramework/AutoSaveScene`

### `XPAutoSave`

特点：

- `[InitializeOnLoad]`
- 静态构造时注册 `EditorApplication.update`
- 定时调用 `EditorSceneManager.SaveScene(nowScene)`

这套机制适合个人开发或原型阶段，目标很明确：防止编辑器里长时间未保存。

## 2. 资源导入规则

目录：

- `Assets/YFramework/Editor/ImportResSeting`

### `AudioSetting`

文件：

- `Assets/YFramework/Editor/ImportResSeting/AudioSetting.cs`

作用：

- 作为 `AssetPostprocessor` 处理导入音频
- 根据音频时长选择 `AudioClipLoadType`

当前策略：

- 小于 1 秒：`DecompressOnLoad`
- 否则：`Streaming`

### `InputResourcesSetting`

文件：

- `Assets/YFramework/Editor/ImportResSeting/InputResourcesSetting.cs`

定位：

- 资源导入命名检查器

当前可见能力：

- 在指定路径下检查导入资源名是否符合正则规则
- 若不符合，记录错误并尝试弹出命名管理窗口

注意：

- 文件中一些贴图导入设置逻辑目前被注释掉了
- 因此它更像一个“预留中的导入规范入口”

### `NameMgrWindow`

文件存在于同目录，用于承接命名检查失败后的编辑器交互。
从整体结构看，它属于资源规范化流程的一部分。

## 3. 通用编辑器工具

目录：

- `Assets/YFramework/Editor/Utility`

### `AssetDataBaseUtility`

定位：

- 对 `AssetDatabase` 相关操作做包装或简化

### `LogUtility`

定位：

- 编辑器日志或输出相关工具

这两个文件的角色更偏“辅助型工具箱”，是编辑器开发时的便捷层，不是主架构入口。

## 4. UI 编辑器支持

目录：

- `Assets/YFramework/UI/Editor`

### `MenuOptions`

文件：

- `Assets/YFramework/UI/Editor/MenuOptions.cs`

作用：

- 在 Unity `GameObject` 菜单下新增 YFramework UI 项
- 自动创建或复用 Canvas
- 自动创建 EventSystem
- 生成自定义 UI 控件对象

当前菜单项包括：

- `GameObject/YFreamwork/UI/PolygonColliderImage`
- `GameObject/YFreamwork/UI/CircleImage`

这说明 YFramework 不只是提供运行时控件，也把控件创建流程编辑器化了。

### `CricleImageEditor`

文件：

- `Assets/YFramework/UI/Editor/CricleImageEditor.cs`

作用：

- 为 `CircleImage` 提供自定义 Inspector
- 在编辑器里暴露 `showPercent` 与 `segments`

这类脚本的价值在于让自定义控件更容易调参数，而不用只依赖默认 Inspector。

## 5. 与运行时代码的协作关系

编辑器工具层和运行时层的关系很清晰：

- 编辑器层负责生成、配置、约束
- 运行时层负责实际表现与逻辑

例如：

- `AutoBindEditor` 生成字段，运行时组件持有这些字段
- `CricleImageEditor` 提供面板，`CircleImage` 真正画图
- 导入设置器规范资源，运行时脚本消费资源

## 6. 当前边界

### 1. 很多工具更偏个人工作流增强

例如自动保存、命名检查、快捷建 UI，都明显带有“提升作者工作效率”的目的。

### 2. 资源导入体系并不算完整

有一些方向已经存在入口，但仍然可以继续扩展，例如：

- 图片导入规范
- 更细颗粒度的命名规则
- 批量修复工具

### 3. 编辑器层分散但职责还算明确

并没有形成一整套统一 Editor Framework，而是按需求放在不同小目录下。

## AI 修改建议

如果你要改编辑器逻辑，优先先判断：

- 是生成代码？优先看 AutoBind
- 是导入资源规则？优先看 `ImportResSeting`
- 是场景工作流问题？优先看 `AutoSaveScene`
- 是 UI 创建体验？优先看 `UI/Editor`

还要特别注意：

- 编辑器脚本的命名空间与运行时脚本不同
- 不要把 `UnityEditor` 依赖带进运行时代码
- 如果修改生成器类，最好同步检查测试样例输出是否还能成立
